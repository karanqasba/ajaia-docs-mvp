# Architecture Note

## Product slice

The assignment asked for a lightweight Google Docs-inspired editor, not a full Google Docs clone. I prioritized the core reviewer-evaluable flow: create, edit, save, upload/import, share, and reopen documents.

## Architecture

The app is a React single-page application using Supabase as the persistence layer.

### Frontend

- React + Vite for fast development and deployment
- TipTap for rich-text editing
- Simple layout with a document list, editor area, save action, import action, and sharing controls

### Persistence

Supabase stores:

- `documents`
  - title
  - rich-text HTML content
  - owner_id
  - timestamps
- `document_shares`
  - document_id
  - owner_id
  - shared_with_user_id

Formatting is persisted as HTML generated by TipTap.

### Users and sharing

To stay inside the timebox, authentication is simulated with seeded users in the UI. This keeps the evaluation focused on document ownership and sharing behavior rather than login plumbing.

The sharing model supports:

- document owner
- grant access to another user
- visible distinction between owned and shared documents

## Prioritization decisions

I prioritized:

1. End-to-end document workflow
2. Persistent storage
3. Clear ownership/shared distinction
4. Basic rich-text editing
5. Simple, testable file import
6. Clear documentation and deployment path

I intentionally deprioritized:

- real-time editing
- Google OAuth or enterprise auth
- complex permissions
- comments/suggestions
- PDF/DOCX import

These were deprioritized because they would add implementation risk without improving the core evaluation as much as a reliable working product slice.
